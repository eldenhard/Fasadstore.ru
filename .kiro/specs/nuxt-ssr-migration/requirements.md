# Документ требований: Миграция Vue 3 SPA → Nuxt SSR/SSG/Hybrid

## Введение

Данный документ описывает требования к миграции существующего коммерческого сайта компании по производству алюминиевых окон и дверей с Vue 3 SPA (Vite + vue-router + GitHub Pages) на Nuxt 4 с гибридным рендерингом (SSR/SSG/CSR). Миграция обусловлена бизнес-необходимостью: органический трафик из Яндекс и Google критически важен для привлечения клиентов, а текущая SPA-архитектура ограничивает SEO-возможности проекта и усложняет достижение целевых показателей индексации, управления метаданными и производительности на уровне Core Web Vitals по сравнению с SSR/SSG-архитектурой.

Целевая платформа: Nuxt 4 (стабильный релиз). Nuxt 3 остаётся поддерживаемым до 31 июля 2026, однако для нового проекта выбирается актуальная ветка Nuxt 4. Целевая UI-система: Nuxt UI v4 (125+ компонентов, Reka UI + Tailwind CSS v4, требует Nuxt 4, обёртка `<UApp>`). Bootstrap 5 выводится из основной UI-архитектуры и сохраняется только как временный legacy-слой на период миграции.

Текущий стек: Vue 3.3 + Vite 5 + TypeScript + Bootstrap 5 + PrimeVue 3 + Swiper 11 + vue-yandex-maps + Pinia. Сайт содержит 6 основных маршрутов (главная, о компании, контакты, объекты, окна с 3 подтипами, двери с 3 подтипами), развёрнут на GitHub Pages.

## Глоссарий

- **Nuxt_App** — целевое приложение на Nuxt 4 с гибридным рендерингом, заменяющее текущий Vue 3 SPA
- **SSR_Engine** — серверный рендеринг Nuxt, генерирующий HTML на сервере при каждом запросе
- **SSG_Engine** — генератор статических страниц Nuxt, создающий HTML на этапе сборки
- **CSR_Mode** — клиентский рендеринг, используемый для интерактивных компонентов после гидратации
- **SEO_Module** — система управления мета-тегами, canonical, robots, Open Graph, schema.org
- **Sitemap_Generator** — модуль генерации XML-карты сайта
- **Image_Optimizer** — система оптимизации изображений (nuxt/image)
- **Router_System** — файловая маршрутизация Nuxt, заменяющая vue-router
- **Layout_System** — система лейаутов Nuxt для общих элементов (navbar, footer)
- **Mobile_Layout** — адаптивная система отображения для мобильных устройств
- **CWV_Monitor** — система мониторинга Core Web Vitals (LCP, CLS, INP)
- **Schema_Generator** — генератор структурированных данных schema.org (JSON-LD)
- **Breadcrumb_System** — система хлебных крошек для навигации и SEO
- **Contact_Form** — форма обратной связи на странице контактов
- **Map_Widget** — виджет Яндекс.Карт на странице контактов
- **Product_Catalog** — каталог продукции (окна, двери) с подтипами
- **Portfolio_Gallery** — галерея выполненных объектов с изображениями
- **UI_System** — целевая UI-система проекта: Nuxt UI v4 (Reka UI + Tailwind CSS v4), заменяющая Bootstrap 5 + PrimeVue 3

## Требования

### Требование 1: Стратегическое решение о миграции и целевая архитектура

**User Story:** Как главный фронтенд-архитектор, я хочу иметь обоснованное решение о миграции на Nuxt с гибридным рендерингом, чтобы обеспечить индексацию сайта поисковыми системами и рост органического трафика.

#### Критерии приёмки

1. THE Nuxt_App SHALL использовать Nuxt 4 в качестве фреймворка с поддержкой гибридного рендеринга (SSR, SSG, CSR на уровне отдельных маршрутов); Nuxt 4 выбран как актуальная целевая платформа, несмотря на продолжающуюся поддержку Nuxt 3 до 31 июля 2026
2. THE Nuxt_App SHALL сохранять совместимость с Vue 3 Composition API, Pinia, TypeScript и существующей бизнес-логикой компонентов
3. THE Nuxt_App SHALL поддерживать развёртывание на платформах с поддержкой серверного рендеринга (Node.js-сервер, Vercel, Netlify или аналогичные); production-деплой SHALL быть перенесён с GitHub Pages на платформу, поддерживающую Nuxt SSR/edge/server execution, так как GitHub Pages является статическим хостингом и несовместим с SSR
4. WHEN выбирается режим рендеринга для маршрута, THE Nuxt_App SHALL назначать SSG для статических страниц (главная, о компании, контакты) и SSG для портфолио объектов; режим рендеринга каталога продукции (окна, двери) SHALL определяться по частоте обновления контента — при статичном контенте приоритет отдаётся SSG/prerender, при динамическом или часто обновляемом — SSR

### Требование 2: Матрица рендеринга по типам страниц

**User Story:** Как главный фронтенд-архитектор, я хочу определить режим рендеринга для каждого типа страницы, чтобы оптимизировать SEO и производительность для каждого маршрута.

#### Критерии приёмки

1. THE Nuxt_App SHALL рендерить маршрут `/` (главная страница) в режиме SSG с полной предгенерацией HTML
2. THE Nuxt_App SHALL рендерить маршрут `/about` (о компании) в режиме SSG с полной предгенерацией HTML
3. THE Nuxt_App SHALL рендерить маршрут `/contacts` (контакты) в режиме SSG для статического контента; все browser-dependent виджеты (Map_Widget, Swiper, vue-scroll-reveal и любые компоненты, зависящие от `window`/`document`) SHALL загружаться на клиенте через `<ClientOnly>` обёртку или клиентские плагины
4. THE Nuxt_App SHALL рендерить маршрут `/objects` (портфолио) в режиме SSG с предгенерацией галереи объектов
5. THE Nuxt_App SHALL рендерить маршруты `/windows` и `/windows/*` (каталог окон) в режиме SSG/prerender при статичном контенте или SSR при динамическом контенте; выбор режима определяется частотой обновления данных каталога
6. THE Nuxt_App SHALL рендерить маршруты `/doors` и `/doors/*` (каталог дверей) в режиме SSG/prerender при статичном контенте или SSR при динамическом контенте; выбор режима определяется частотой обновления данных каталога
7. WHEN поисковый робот запрашивает любую страницу, THE Nuxt_App SHALL возвращать полностью отрендеренный HTML с контентом, мета-тегами и структурированными данными

### Требование 3: Техническая SEO-архитектура

**User Story:** Как главный фронтенд-архитектор, я хочу внедрить полноценную SEO-систему, чтобы сайт корректно индексировался Яндексом и Google и занимал высокие позиции по целевым запросам.

#### Критерии приёмки

1. THE SEO_Module SHALL генерировать уникальные теги `<title>` и `<meta name="description">` для каждого маршрута
2. THE SEO_Module SHALL генерировать тег `<link rel="canonical">` с абсолютным URL для каждой страницы
3. THE SEO_Module SHALL генерировать тег `<meta name="robots">` со значением `index, follow` для всех публичных страниц
4. THE SEO_Module SHALL генерировать Open Graph теги (`og:title`, `og:description`, `og:image`, `og:url`, `og:type`, `og:locale`) для каждой страницы
5. THE Sitemap_Generator SHALL генерировать файл `/sitemap.xml` со всеми публичными маршрутами, включая вложенные маршруты окон и дверей
6. THE Nuxt_App SHALL предоставлять файл `/robots.txt` с директивами для Yandex и Googlebot, включая ссылку на sitemap.xml
7. THE Schema_Generator SHALL генерировать JSON-LD разметку schema.org типа `Organization` на главной странице с данными компании
8. THE Schema_Generator SHALL генерировать JSON-LD разметку schema.org типа `Product` на страницах каталога окон и дверей
9. THE Schema_Generator SHALL генерировать JSON-LD разметку schema.org типа `BreadcrumbList` на всех страницах с глубиной вложенности более 1
10. THE Breadcrumb_System SHALL отображать навигационные хлебные крошки на всех страницах кроме главной
11. THE Nuxt_App SHALL использовать семантическую HTML-структуру с одним `<h1>` на страницу и иерархией заголовков `<h2>`–`<h6>`
12. THE Nuxt_App SHALL устанавливать атрибут `lang="ru"` в корневом теге `<html>` для всех страниц

### Требование 4: Mobile-First адаптивная стратегия

**User Story:** Как главный фронтенд-архитектор, я хочу реализовать полноценную mobile-first адаптивную стратегию, чтобы сайт обеспечивал качественный пользовательский опыт на всех устройствах.

#### Критерии приёмки

1. THE Mobile_Layout SHALL использовать mobile-first подход с брейкпоинтами: 320px (mobile), 768px (tablet), 1024px (desktop), 1440px (wide)
2. THE Nuxt_App SHALL обеспечивать минимальный размер интерактивных элементов (кнопки, ссылки) 44×44 CSS-пикселей на мобильных устройствах
3. THE Nuxt_App SHALL предотвращать сдвиги макета (CLS) путём указания явных размеров для изображений, слайдеров и встраиваемых элементов
4. THE Mobile_Layout SHALL адаптировать навигацию: полное меню на десктопе, гамбургер-меню на мобильных устройствах с шириной экрана менее 1024px
5. THE Mobile_Layout SHALL обеспечивать корректное отображение таблиц на мобильных устройствах через горизонтальную прокрутку или перестроение в карточки
6. THE Nuxt_App SHALL использовать CSS `env(safe-area-inset-*)` для корректного отображения на устройствах с вырезами экрана
7. THE Contact_Form SHALL адаптировать поля ввода до полной ширины экрана на мобильных устройствах с шириной менее 768px
8. THE Mobile_Layout SHALL обеспечивать корректное отображение модальных окон на мобильных устройствах без выхода за границы viewport
9. WHEN пользователь просматривает Product_Catalog на мобильном устройстве, THE Mobile_Layout SHALL отображать карточки продукции в одну колонку
10. THE Mobile_Layout SHALL использовать адаптивную типографику с базовым размером шрифта 16px и масштабированием через `clamp()` или `rem`

### Требование 5: Core Web Vitals и производительность

**User Story:** Как главный фронтенд-архитектор, я хочу оптимизировать Core Web Vitals, чтобы сайт соответствовал требованиям Google и Яндекс для ранжирования.

#### Критерии приёмки

1. THE Nuxt_App SHALL обеспечивать показатель LCP (Largest Contentful Paint) менее 2.5 секунд на мобильных устройствах
2. THE Nuxt_App SHALL обеспечивать показатель CLS (Cumulative Layout Shift) менее 0.1 на всех страницах
3. THE Nuxt_App SHALL обеспечивать показатель INP (Interaction to Next Paint) менее 200 миллисекунд
4. THE Image_Optimizer SHALL конвертировать изображения в формат WebP или AVIF и генерировать адаптивные размеры через `<picture>` или `srcset`
5. THE Image_Optimizer SHALL применять ленивую загрузку (`loading="lazy"`) для изображений за пределами первого экрана
6. THE Nuxt_App SHALL загружать шрифты с использованием `font-display: swap` и предзагрузки критических шрифтов через `<link rel="preload">`
7. THE Nuxt_App SHALL применять code splitting на уровне маршрутов для минимизации размера начального JavaScript-бандла
8. THE Nuxt_App SHALL обеспечивать оптимизацию критического пути рендеринга и минимизацию render-blocking CSS для ускорения первой отрисовки
9. THE Nuxt_App SHALL загружать browser-dependent и legacy client-only зависимости (Яндекс.Карты, Swiper и временно сохраняемые legacy UI-модули) асинхронно или по требованию
10. WHEN происходит гидратация на клиенте, THE Nuxt_App SHOULD предусматривать возможность отложенной или частичной гидратации для некритичных компонентов за пределами viewport

### Требование 6: Файловая структура и маршрутизация Nuxt

**User Story:** Как главный фронтенд-архитектор, я хочу определить production-ready файловую структуру Nuxt 4-проекта, чтобы обеспечить масштабируемость и поддерживаемость кодовой базы.

#### Критерии приёмки

1. THE Router_System SHALL использовать файловую маршрутизацию Nuxt в директории `app/pages/` для всех маршрутов, заменяя ручную конфигурацию vue-router; Nuxt 4 использует директорию `app/` как корень приложения
2. THE Router_System SHALL поддерживать вложенные маршруты через структуру директорий: `app/pages/windows/classic.vue`, `app/pages/windows/panorama.vue`, `app/pages/windows/hidden.vue`, `app/pages/doors/hd.vue`, `app/pages/doors/balcony.vue`, `app/pages/doors/classic.vue`
3. THE Layout_System SHALL определять лейаут `default.vue` в директории `app/layouts/` с общими элементами (navbar, footer) на Nuxt UI
4. THE Nuxt_App SHALL размещать переиспользуемые компоненты в директории `app/components/` с автоимпортом Nuxt
5. THE Nuxt_App SHALL размещать composables в директории `app/composables/` с автоимпортом Nuxt
6. THE Nuxt_App SHALL размещать серверные утилиты и API-эндпоинты в директории `server/`
7. THE Nuxt_App SHALL размещать статические ресурсы (favicon) в директории `public/` и обрабатываемые ресурсы (изображения, шрифты) в директории `app/assets/`
8. THE Nuxt_App SHALL хранить конфигурацию в файле `nuxt.config.ts` с настройками модулей, рендеринга и SEO

### Требование 7: Переиспользуемая SEO-система (composables)

**User Story:** Как главный фронтенд-архитектор, я хочу создать систему переиспользуемых composables для SEO, чтобы унифицировать управление мета-данными на всех страницах.

#### Критерии приёмки

1. THE SEO_Module SHALL использовать официальный Nuxt composable `useSeoMeta()` для установки title, description, Open Graph и robots-тегов на каждой странице
2. THE SEO_Module SHALL предоставлять внутренний проектный composable `useCanonical()` для генерации canonical URL на основе текущего маршрута
3. THE Schema_Generator SHALL предоставлять внутренний проектный composable `useSchemaOrg()` для генерации JSON-LD разметки с типами Organization, Product, BreadcrumbList
4. THE Breadcrumb_System SHALL предоставлять внутренний проектный composable `useBreadcrumbs()` для автоматической генерации хлебных крошек на основе текущего маршрута
5. THE SEO_Module SHALL предоставлять внутренний проектный composable `useRobots()` для управления директивами robots на уровне отдельных страниц
6. WHEN composable `useSeoMeta()` вызывается без параметров, THE SEO_Module SHALL использовать значения по умолчанию из конфигурации сайта
7. THE SEO_Module SHALL явно разделять в документации и коде: (a) официальные средства Nuxt (`useSeoMeta`, `useHead`, `definePageMeta`) и (b) внутренние проектные composables (`useCanonical`, `useSchemaOrg`, `useBreadcrumbs`, `useRobots`)

### Требование 8: Конфигурация Nuxt и интеграция модулей

**User Story:** Как главный фронтенд-архитектор, я хочу определить конфигурацию nuxt.config.ts с интеграцией необходимых модулей, чтобы обеспечить работу SSR, SEO и оптимизации.

#### Критерии приёмки

1. THE Nuxt_App SHALL настраивать `routeRules` в `nuxt.config.ts` для определения режима рендеринга (SSR/SSG/CSR) для каждой группы маршрутов
2. THE Nuxt_App SHALL интегрировать модуль `@nuxtjs/sitemap` для автоматической генерации sitemap.xml
3. THE Nuxt_App SHALL интегрировать модуль `@nuxt/image` для оптимизации изображений с поддержкой WebP и адаптивных размеров
4. THE Nuxt_App SHALL интегрировать модуль `@nuxtjs/robots` для управления файлом robots.txt
5. THE Nuxt_App SHALL настраивать глобальные SEO-метаданные по умолчанию в секции `app.head` конфигурации; page-level overrides SHALL выполняться через официальные composables `useSeoMeta()` / `useHead()`
6. THE Nuxt_App SHALL использовать Nuxt UI v4 (`@nuxt/ui`) как основной UI-модуль; глобальная стилизация SHALL быть выстроена на Nuxt UI + Tailwind CSS v4; SCSS может использоваться точечно для legacy-стилей или специфичных кастомных блоков на переходном этапе
7. THE Nuxt_App SHALL оборачивать приложение в компонент `<UApp>` в `app/app.vue`, так как это требуется для корректной работы Toast, Tooltip и programmatic overlays в Nuxt UI v4
8. THE Nuxt_App SHALL настраивать TypeScript в строгом режиме
9. IF модуль не поддерживает SSR, THEN THE Nuxt_App SHALL загружать модуль только на клиенте через `<ClientOnly>` обёртку или плагин с режимом `client`

### Требование 9: Поэтапный план миграции

**User Story:** Как главный фронтенд-архитектор, я хочу иметь поэтапный план миграции с оценкой рисков, чтобы минимизировать простой и обеспечить плавный переход.

#### Критерии приёмки

1. THE Nuxt_App SHALL мигрироваться в 5 фаз: (1) инфраструктура и конфигурация Nuxt 4, (2) лейауты и общие компоненты на Nuxt UI, (3) страницы и маршрутизация, (4) SEO, оптимизация и QA, (5) вывод legacy-зависимостей (Bootstrap, PrimeVue)
2. WHEN выполняется фаза 1, THE Nuxt_App SHALL включать: инициализацию Nuxt 4-проекта, настройку nuxt.config.ts, подключение модулей (@nuxt/ui, @nuxt/image, @nuxtjs/sitemap, @nuxtjs/robots), настройку TypeScript и Tailwind CSS v4
3. WHEN выполняется фаза 2, THE Nuxt_App SHALL включать: создание default layout с navbar и footer на Nuxt UI, миграцию UI-компонентов (кнопки, формы, модальные окна) с Bootstrap/PrimeVue на Nuxt UI, миграцию composables
4. WHEN выполняется фаза 3, THE Nuxt_App SHALL включать: создание файловой структуры app/pages/, миграцию каждой страницы с сохранением функциональности и контента, настройку вложенных маршрутов для окон и дверей
5. WHEN выполняется фаза 4, THE Nuxt_App SHALL включать: внедрение SEO-composables, настройку schema.org, оптимизацию изображений, тестирование Core Web Vitals, настройку sitemap и robots.txt, проверку 301-редиректов и content parity
6. WHEN выполняется фаза 5, THE Nuxt_App SHALL включать: полный вывод Bootstrap 5 CSS/JS, полный вывод PrimeVue 3, удаление неиспользуемых legacy-зависимостей, финальную проверку bundle size
7. IF при миграции компонента обнаруживается несовместимость с SSR, THEN THE Nuxt_App SHALL оборачивать компонент в `<ClientOnly>` и документировать причину

### Требование 10: Миграция сторонних библиотек

**User Story:** Как главный фронтенд-архитектор, я хочу определить стратегию миграции каждой сторонней библиотеки, чтобы обеспечить совместимость с Nuxt SSR и консистентность с целевой UI-системой.

#### Критерии приёмки

1. THE Nuxt_App SHALL использовать Nuxt UI v4 как основную UI-систему; Bootstrap 5 CSS SHALL использоваться только как временный legacy-слой на этапе миграции для компонентов, которые ещё не перенесены на Nuxt UI; Bootstrap JS SHALL загружаться только на клиенте через плагин с режимом `client` до полного вывода
2. THE Nuxt_App SHALL заменять PrimeVue 3 компоненты (InputText, Button, Toast) на эквиваленты из Nuxt UI v4; PrimeVue SHALL сохраняться только для тех компонентов, которые реально невыгодно переписывать на первом этапе, и SHALL быть полностью выведен к завершению миграции
3. THE Nuxt_App SHALL загружать Swiper 11 через `<ClientOnly>` обёртку или динамический импорт с `ssr: false`
4. THE Nuxt_App SHALL загружать vue-yandex-maps через `<ClientOnly>` обёртку, так как Яндекс.Карты API требует браузерного окружения
5. THE Nuxt_App SHALL заменять vue-scroll-reveal на Nuxt-совместимую альтернативу или загружать через клиентский плагин, так как библиотека зависит от DOM
6. THE Nuxt_App SHALL мигрировать Pinia store без изменений, используя встроенную интеграцию Pinia в Nuxt 4
7. IF сторонняя библиотека вызывает ошибку `window is not defined` или `document is not defined` при SSR, THEN THE Nuxt_App SHALL изолировать библиотеку в клиентском контексте
8. THE Nuxt_App SHALL не допускать одновременного использования более двух UI-библиотек в production; целевое состояние — только Nuxt UI v4 как единственная UI-система

### Требование 11: QA, валидация и мониторинг

### Требование 12: Риски и типичные ошибки миграции SPA → Nuxt

**User Story:** Как главный фронтенд-архитектор, я хочу знать типичные риски и ошибки при миграции SPA на Nuxt, чтобы избежать распространённых проблем.

#### Критерии приёмки

1. IF компонент обращается к `window`, `document` или `navigator` в серверном контексте, THEN THE Nuxt_App SHALL защищать обращение проверкой `import.meta.client` или `<ClientOnly>`
2. IF маршрут использует `keep-alive` (как текущий ObjectsView), THEN THE Nuxt_App SHALL проверять совместимость `keep-alive` с SSR и при необходимости заменять на клиентское кэширование
3. IF компонент использует `defineAsyncComponent`, THEN THE Nuxt_App SHALL заменять на встроенный механизм ленивой загрузки Nuxt (префикс `Lazy` для компонентов)
4. IF глобальные стили Bootstrap конфликтуют с Nuxt-лейаутами, THEN THE Nuxt_App SHALL изолировать стили через scoped CSS или CSS-модули
5. IF Яндекс.Карты API загружается до готовности DOM, THEN THE Map_Widget SHALL инициализировать карту только после события `onMounted` внутри `<ClientOnly>`
6. IF изображения в Portfolio_Gallery не имеют явных размеров, THEN THE Image_Optimizer SHALL добавлять атрибуты `width` и `height` для предотвращения CLS

### Требование 13: Итоговые рекомендации по приоритетам

2. THE Nuxt_App SHALL включать в категорию «Обязательно»: миграцию на Nuxt 4, SSR/SSG рендеринг, SEO мета-теги, sitemap, robots.txt, schema.org (Organization), mobile-first адаптивность, оптимизацию изображений, внедрение Nuxt UI v4 как основной UI-системы для всех новых и перерабатываемых страниц, перенос деплоя с GitHub Pages на SSR-совместимую платформу, сохранение URL-структуры и 301-редиректы

### Требование 14: Redirect Map и сохранение URL-структуры

**User Story:** Как главный фронтенд-архитектор, я хочу обеспечить сохранение URL-структуры и настроить 301-редиректы при миграции, чтобы не потерять накопленный SEO-вес и не допустить появления 404-ошибок для проиндексированных страниц.

#### Критерии приёмки

1. THE Nuxt_App SHALL сохранять текущую URL-структуру всех маршрутов (`/`, `/about`, `/contacts`, `/objects`, `/windows/*`, `/doors/*`) без изменений
2. IF URL-структура маршрута изменяется при миграции, THEN THE Nuxt_App SHALL настраивать 301-редирект со старого URL на новый через `routeRules` или серверный middleware
3. THE Nuxt_App SHALL предоставлять полную карту редиректов (redirect map) в виде конфигурации, документирующей все изменения URL

### Требование 15: Content Parity (паритет контента)

**User Story:** Как главный фронтенд-архитектор, я хочу гарантировать, что при миграции ни один индексируемый контент не будет потерян, чтобы сохранить SEO-позиции и пользовательский опыт.

### Требование 16: Release Strategy (стратегия выпуска)

**User Story:** Как главный фронтенд-архитектор, я хочу иметь чёткую стратегию выпуска мигрированного сайта, чтобы минимизировать риски при переключении production-трафика.

### Требование 18: Целевая UI-система и стратегия миграции UI

**User Story:** Как главный фронтенд-архитектор, я хочу зафиксировать целевую UI-систему проекта и определить стратегию перехода с Bootstrap 5 + PrimeVue 3 на Nuxt UI v4, чтобы обеспечить консистентность, производительность и удобство mobile-first разработки.

#### Критерии приёмки

1. THE Nuxt_App SHALL использовать Nuxt UI v4 (`@nuxt/ui`) как основную и единственную целевую UI-систему проекта; Nuxt UI v4 устанавливается вместе с Tailwind CSS v4 и использует Reka UI как headless-слой с встроенной accessibility-поддержкой
2. THE Nuxt_App SHALL использовать Tailwind CSS v4 как базовый слой токенов, utility-классов и responsive-архитектуры
3. Bootstrap 5 SHALL рассматриваться как временный legacy-слой и использоваться только для компонентов, которые ещё не перенесены на Nuxt UI; Bootstrap SHALL быть полностью выведен из проекта по завершении миграции
4. PrimeVue 3 SHALL сохраняться только для тех компонентов, которые реально невыгодно переписывать на первом этапе; PrimeVue SHALL быть полностью заменён на Nuxt UI по завершении миграции
5. Все новые страницы, новые компоненты и перерабатываемые mobile-first блоки SHALL реализовываться на Nuxt UI, а не на Bootstrap или PrimeVue
6. THE Nuxt_App SHALL определять следующий порядок приоритетов замены legacy-компонентов на Nuxt UI: (1) навигация и header/footer, (2) формы и поля ввода, (3) кнопки и интерактивные элементы, (4) карточки каталога, (5) модальные окна / drawer / mobile overlays, (6) прочие legacy-компоненты
7. THE Nuxt_App SHALL не допускать одновременного использования более двух UI-библиотек на одной странице; на переходном этапе допускается сосуществование Nuxt UI + одной legacy-библиотеки (Bootstrap или PrimeVue) на уровне отдельных страниц
8. Конечное production-состояние SHALL использовать только Nuxt UI v4 как единственную UI-систему; Bootstrap 5 и PrimeVue 3 не должны присутствовать в production bundle после завершения миграции
